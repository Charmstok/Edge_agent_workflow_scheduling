# Edge Agent Workflow Scheduling

An experimental framework for edge workflow scheduling in multi-agent LLM systems.

This project studies how to schedule both LLM inference steps and long-running tool calls generated by multiple LLM agents. The target environment is a pure edge setup with heterogeneous LLM runtimes and tool workers: an Ubuntu server may host a larger local model, other edge boards may host smaller local models, and Raspberry Pi devices run tool workers. The focus is not on implementing OCR, PDF parsing, or table extraction algorithms from scratch. Instead, the project builds standardized workflow-step abstractions, worker and LLM-runtime interfaces, scheduling policies, execution tracing, and a reinforcement-learning optimization loop.

## Goal

The first milestone implements a minimal end-to-end execution loop:

```text
Simulated Agent -> LLMCall / ToolCall -> Scheduler
  -> Local LLM Instance / Local Worker -> Result -> Trace Log
```

The current implementation includes simulated agents, a unified in-memory workflow
queue, baseline scheduling policies, mock LLM runtimes, local tool workers, an
image preprocessing tool, and JSONL trace logging.

## Environment

This project uses `uv` to manage Python environments and dependencies.

## Quick Start

Clone the repository and enter the project directory:

```bash
git clone <repo-url>
cd edge-agent-workflow-scheduling
```

Install development dependencies:

```bash
uv sync --dev
```

Verify that the package can be imported:

```bash
uv run python -c "import edge_agent_workflow_scheduling; print(edge_agent_workflow_scheduling.__name__)"
```

Run tests:

```bash
uv run pytest
```

## Usage

Run the first local end-to-end demo:

```bash
uv run python scripts/run_first_demo.py --policy round_robin
```

The demo creates two simulated agents, two mock LLM runtimes, and two local
workers. It generates LLM inference steps and image preprocessing tool calls,
schedules them through the baseline scheduler, writes JSONL traces under
`data/traces/`, and prints latency, success-rate, worker-count, and
LLM-runtime-count summaries.

Generated demo data lives under `data/` and is ignored by git. See
[`scripts/first_demo_data.md`](scripts/first_demo_data.md) for the generated data
layout and trace fields.

## Configuration

Configuration files live under `configs/`:

- `configs/system.yaml`: system-level settings
- `configs/devices.yaml`: worker device settings
- `configs/rl.yaml`: reinforcement-learning settings
